# Telemetry & Alerting Backend

A high-performance backend microservice designed for telemetry ingestion and real-time rule-based alerting. Built meticulously with Node.js, Express, TypeScript, and MongoDB to meet production-level scale.

## 🏗 Architecture Decisions

The system is strictly built using the **3-Tier Architecture** pattern (`Controller → Service → Repository`). This ensures maximum separation of concerns, scalability, and testability:
- **Controllers**: Handle HTTP protocols, request parsing, and response mapping.
- **Services**: House the pure business logic and rule orchestration.
- **Repositories**: Encapsulate all database queries and indexing logics, isolating Mongoose references entirely from the business logic.

**Additional Patterns:**
- **Centralized Error Handling**: A global middleware catches structural and business-logic errors cleanly without leaking internal traces.
- **Environment Driven**: Isolated configuration patterns dynamically adapt database clusters and listening ports.
- **Idempotency**: Strict validation prevents duplicate metric creation for identical `deviceId` and `timestamp`.

## 📊 Indexing Strategy

Designed explicitly for write-heavy workloads transitioning into dynamic read patterns:

1. **Telemetry Idempotency & Lookups**:
   - `Compound Unique Index`: `{ deviceId: 1, sourceId: 1, timestamp: 1 }` uniquely guarantees complete idempotency preventing any duplicated records on the exact exact parameters.
   - `Performance Compound Index`: `{ deviceId: 1, timestamp: -1 }` explicitly crafted to resolve the highly requested `getLatest()` and time-range history queries optimally.

2. **Alert Duplication Locks**:
   - `Partial Unique Index`: `{ deviceId: 1, rule: 1, status: 1 }` with a `partialFilterExpression: { status: "OPEN" }` guarantees that a device can never have two globally "OPEN" alerts generated by the exact same rule, while still allowing historical tracking for ACKNOWLEDGED/RESOLVED instances.

## 🧠 Rule Engine Design

The engine is engineered as a lightweight, functional, purely functional in-memory abstraction pipeline (`src/modules/rules/rule.engine.ts`):
1. **Pipelining**: Instead of tightly coupling rules to the DB, it statically loads raw telemetry and an injected array of recent `history` arrays directly from the controller injection.
2. **Stateless Mapping**: The rules validate single instances (e.g. `metricA > 100`) natively.
3. **Stateful Sequence Tracking**: Historical consecutive anomalies (e.g. `metricC === 0`) are validated by mathematically slicing the trailing history parameters. Every iteration is $O(1)$ relative memory overhead per request.

## ⚡ API Endpoints reference

**Telemetry Input:**
- `POST /api/v1/telemetry`: Send continuous streams

**Monitoring:**
- `GET /api/v1/devices/:id/latest`: Retrieve instant status
- `GET /api/v1/devices/:id/history?from=&to=`: Retrieve historical streams
- `GET /api/v1/alerts?status=OPEN`: Monitor triggered states

## 📦 How to Run

1. `npm install`
2. Configure `.env` (optional, default provided)
3. `npm run dev` to start locally.
